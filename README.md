## 🤖 WHAT IN THE CIRCUIT BOARD IS CPR?

Picture this: It's 2026, and somewhere in the vast digital expanse of the Solana blockchain, a piece of software has achieved something remarkable. Not consciousness exactly (the lawyers asked us to emphasize this), but something... interesting. **CPR** is an autonomous token deployment system that doesn't just follow orders—it makes decisions, manages resources, and deploys tokens to the blockchain while you sleep, eat, or contemplate the philosophical implications of giving financial autonomy to an algorithm.

Born from the fever dream of developers who asked themselves "What if a bot could deploy tokens based on community votes without us having to click buttons?", CPR has evolved into a sophisticated system that's part democracy engine, part blockchain automaton, and part retro-futuristic fever dream. It's like if DOOM (the 1993 first-person shooter, not the Marvel villain) had a baby with a DAO, and that baby grew up to become a responsible financial entity with a penchant for deploying tokens every 20 minutes.

### 🎯 THE CORE PHILOSOPHY

At its heart, CPR believes in a simple principle: **Communities should decide what gets built, and automation should handle the boring parts.** Why should token deployment require manual intervention, sleep deprivation, and frantically clicking buttons? Why should communities wait for developers to "find time" to deploy their voted-upon ideas? CPR said "not on my watch" (it doesn't have a watch, but you get the idea) and automated the entire pipeline from suggestion to deployment.

### ✨ WHAT MAKES CPR SPECIAL?

**🎯 Fully Autonomous Token Deployment**
CPR doesn't need permission. It doesn't need approval from a committee. It doesn't need someone to wake up and press the "deploy" button. When the timer hits zero and the community has spoken through their votes, CPR springs into action like a caffeinated squirrel with blockchain access. It evaluates suggestions, selects winners based on democratic voting, generates professional token metadata, deploys to Solana, and updates all records—all without human intervention.

**🧠 Democratic Decision-Making Engine**
Forget centralized control and benevolent dictators. CPR implements true democracy where every vote counts, every suggestion gets fair consideration, and the community collectively decides what gets deployed next. It's democracy, but with better response times and fewer committee meetings.

**💎 Autonomous Treasury Management**
CPR manages its own treasury with the financial discipline of a seasoned CFO who's had three espressos and a financial modeling bootcamp. It tracks balances, manages deployment costs, handles transaction fees, and does it all while maintaining full transparency. Some human CFOs should take notes.

**📊 Real-Time Analytics & Monitoring**
Knowledge is power, and CPR provides more metrics than a fitness tracker on a data scientist. Track deployments in real-time, monitor community engagement, watch the treasury balance, and get insights into every aspect of the system's operation through a sleek, retro-futuristic interface.

**🎨 Dynamic Metadata Generation**
Every token deployed by CPR isn't just another SPL token—it's a fully-realized blockchain entity with custom metadata, unique identifiers, professional descriptions, and all the documentation needed to make it legitimate. CPR treats every deployment like it's launching the next big thing (because who knows, maybe it is).

**🚀 Never Sleeps, Never Quits**
Unlike humans with their pesky needs for sleep, food, and Netflix binges, CPR operates 24/7/365. It's always watching, always ready, always prepared to deploy the next community-selected token. It's the employee of the month every month, except it's a bot and doesn't care about your pizza parties.

### 📥 INSTALLATION GUIDE

Now comes the fun part. Follow these steps carefully, and you'll have CPR running on your machine faster than you can say "autonomous token deployment system" three times fast.

#### STEP 1: CLONE THE REPOSITORY

First, let's get the code onto your machine. Open your terminal (that black window with text that makes you look like a hacker) and run:

```bash
git clone https://github.com/CPROfficial/CPR.git
cd CPR
```

**What's happening:** You're downloading the entire CPR codebase from GitHub and navigating into the project directory. Git is doing all the heavy lifting here, pulling down every file, every component, every ASCII art character we've lovingly placed.

**Pro tip:** If you want to contribute later, fork the repository first and clone your fork instead.

#### STEP 2: INSTALL DEPENDENCIES

Now we need to grab all the third-party packages that CPR depends on. Think of this as gathering ingredients before cooking a meal:

```bash
npm install
```

*Go grab that coffee we mentioned earlier. Depending on your internet connection and npm's mood, this could take anywhere from 30 seconds to "why is it taking so long, should I restart?"*

**What's happening:** npm is reading the `package.json` file, identifying all required dependencies (React, TypeScript, Supabase client, etc.), and downloading them into the `node_modules` folder. You'll end up with thousands of files, most of which you'll never directly interact with, but all of which are somehow important.

**Pro tip:** If you encounter errors, try `npm cache clean --force` and run `npm install` again. This solves 73% of npm-related problems (statistic made up, but probably accurate).

#### STEP 3: SET UP YOUR ENVIRONMENT VARIABLES

Create a `.env` file in the root directory of the project. This file will contain your secret configuration that should never, ever be committed to Git:

```bash
# Copy the example env file if it exists, or create new one
cp .env.example .env  # or just: touch .env
```

Now edit the `.env` file with your favorite text editor and add:

```bash
VITE_SUPABASE_URL=your_supabase_project_url_here
VITE_SUPABASE_ANON_KEY=your_supabase_anonymous_key_here
```

**Where to get these values:**
1. Log into your Supabase dashboard at https://app.supabase.com
2. Select your project (or create a new one if you haven't already)
3. Go to Settings → API
4. Copy the "Project URL" for `VITE_SUPABASE_URL`
5. Copy the "anon public" key for `VITE_SUPABASE_ANON_KEY`

**What's happening:** These environment variables tell CPR where your Supabase database lives and how to authenticate with it. The `VITE_` prefix is important—it tells Vite to make these variables available in the browser bundle.

**Pro tip:** NEVER commit your `.env` file to Git. Ever. The `.gitignore` file should already be configured to ignore it, but double-check. Exposed keys are how security incidents happen, and security incidents are how sad stories begin.

#### STEP 4: SET UP THE DATABASE

CPR requires a specific database schema with tables, policies, and all the fun PostgreSQL stuff. Fortunately, we've prepared migrations that handle everything:

**Option A: Using Supabase CLI (Recommended)**

```bash
# Install Supabase CLI if you haven't already
npm install -g supabase

# Link your project
supabase link --project-ref your-project-ref

# Push all migrations
supabase db push
```

**Option B: Using the Supabase Dashboard**

1. Navigate to your Supabase project dashboard
2. Go to the SQL Editor
3. Open each migration file from `supabase/migrations/`
4. Copy the SQL and run it in order (they're timestamped, go in order)

**What's happening:** The migrations create several tables:
- `admin_settings`: Stores configuration like deployment intervals and wallet addresses
- `suggestions`: Stores community token suggestions and their vote counts
- `tokens`: Records every deployed token with metadata and blockchain addresses
- Row Level Security policies on everything (because security matters)

**Pro tip:** If you mess up the database and want to start fresh, you can always drop all tables and re-run migrations. Just don't do this in production (you shouldn't have production access anyway if you're reading installation instructions).

#### STEP 5: DEPLOY EDGE FUNCTIONS

CPR uses serverless Edge Functions for autonomous operations. Let's deploy them:

```bash
# Deploy the token generation function
supabase functions deploy generate-token --project-ref your-project-ref

# Deploy the admin operations function
supabase functions deploy admin-operations --project-ref your-project-ref
```

**What's happening:** These functions are being uploaded to Supabase's edge network where they'll run autonomously. The `generate-token` function handles the entire deployment process when timers expire, while `admin-operations` handles configuration changes and manual overrides.

**Pro tip:** Edge Functions can take a minute or two to propagate across the network. If they don't work immediately after deployment, give it a moment.

Run this SQL in the Supabase SQL Editor.

**What's happening:** You're creating the master configuration record that CPR will reference for all deployment operations. The UUID is hardcoded (so we always know where to find it), and the wallet address is where deployment fees will be paid from.

### 🎯 FEATURE 1: AUTONOMOUS TOKEN DEPLOYMENT

This is the crown jewel, the piece de resistance, the reason CPR exists. Autonomous token deployment means **CPR handles the entire process from suggestion selection to blockchain deployment without requiring a human to click a single button**.

#### How It Works (In Excruciating Detail)

CPR operates on a timer-based system. You configure an interval (say, 20 minutes), and every 20 minutes, CPR springs into action like a caffeinated clockwork machine:

1. **Timer Expiration**: The deployment timer hits zero
2. **Check Autonomous Mode**: Verifies that automatic deployments are enabled (safety first!)
3. **Fetch Suggestions**: Queries the database for all suggestions with status "approved"
4. **Apply Filters**: Removes suggestions that don't meet minimum vote thresholds
5. **Sort by Votes**: Orders suggestions by total votes (highest first)
6. **Select Winner**: Takes the top suggestion as the deployment candidate
7. **Generate Metadata**: Creates comprehensive token metadata including name, symbol, description, and imagery
8. **Deploy to Blockchain**: Creates the SPL token on Solana with all metadata
9. **Record Deployment**: Updates the database with deployment information
10. **Update Suggestion**: Marks the winning suggestion as "deployed"
11. **Reset Timer**: Starts the countdown for the next deployment
12. **Broadcast Update**: Real-time notifications inform all connected clients
13. **Profit** (metaphorically): Everyone celebrates

### 🗳️ FEATURE 2: COMMUNITY VOTING SYSTEM

Democracy is messy, but blockchain democracy? That's a whole different ball game. CPR implements a sophisticated voting system that lets communities collectively decide which token ideas deserve to become reality.

#### The Voting Mechanics

**Submitting Suggestions**:
Users can submit token ideas through the Feedback tab. Each suggestion includes:
- Token name (the fun part)
- Token symbol (usually 3-5 characters)
- Description (sell your vision!)
- Optional logo/imagery URL
- Category tags for organization

**Voting**:
Once submitted, suggestions enter the voting arena where community members can:
- **Upvote**: Express support for an idea (+1 to vote count)
- **Downvote**: Express skepticism (−1 to vote count)
- **Comment**: Provide feedback and suggestions
- **Share**: Spread the word to gather more votes

**Vote Counting**:
CPR uses a simple but effective counting mechanism:
```
Total Score = Upvotes − Downvotes
```

No complex algorithms, no vote weighting by wallet size, no plutocracy—just pure democratic counting. One user, one vote (per suggestion).

### 📜 ROW LEVEL SECURITY EXAMPLES

Here's how CPR implements granular access control:

```sql
-- Example 1: Public can view approved suggestions only
CREATE POLICY "public_view_approved_suggestions"
  ON suggestions FOR SELECT
  USING (status = 'approved');

-- Example 2: Authenticated users can submit suggestions
CREATE POLICY "authenticated_can_submit"
  ON suggestions FOR INSERT
  TO authenticated
  WITH CHECK (true);

-- Example 3: Users can only vote once per suggestion
CREATE POLICY "one_vote_per_suggestion"
  ON suggestion_votes FOR INSERT
  TO authenticated
  WITH CHECK (
    NOT EXISTS (
      SELECT 1 FROM suggestion_votes
      WHERE suggestion_id = suggestion_votes.suggestion_id
      AND user_id = auth.uid()
    )
  );

-- Example 4: Only admins can modify admin_settings
CREATE POLICY "only_admins_can_update"
  ON admin_settings FOR UPDATE
  TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM admin_users
      WHERE user_id = auth.uid()
    )
  );
```
